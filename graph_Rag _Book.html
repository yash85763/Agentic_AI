Chapter One

Why Knowledge Graphs

A defence of nodes, edges, and the meaning between them

Most of what we know cannot be compressed into a row in a spreadsheet. It lives in relationships — Pierre Curie married Marie Sklodowska, they co-discovered radium, radium is a chemical element, chemical elements obey the periodic law Mendeleev assembled. Write all of that in a sentence and you have prose; write it in a table and you have four disconnected facts; write it in a graph and you have a structure where the relationships themselves are first-class citizens, queryable, traversable, and extensible without changing the schema.

The Alan Turing Institute's Knowledge Graphs Interest Group, established in 2020 and spanning universities from Edinburgh to Oxford to King's College London, defines the practical promise concisely: knowledge graphs organise data from multiple sources, capture information about entities of interest, and forge connections between them.

Turing Institute · Three Properties of a Knowledge Graph

Graph: unlike traditional knowledge bases, the content of a KG is organised as a graph where nodes, relationships, and attributes are equally important. This enables integration of new datasets and exploration by navigating from one part of the graph to another through links.

Semantic: the meaning of the data is encoded in an ontology — a schema sub-graph describing entity types and their characteristics — allowing the graph to both store data and reason about it.

Alive: KGs are flexible in the types of data and schemas they support; they evolve to reflect changes in the domain, and new data is added as it becomes available.

In AI applications, the Turing group notes three complementary roles: KGs reduce the need for large labelled datasets by encoding domain knowledge explicitly; they facilitate transfer learning and explainability; and they encode knowledge that would be costly to learn from data alone. Against a language model that learns everything implicitly, a knowledge graph is its memory made explicit and auditable.

The remainder of this guide is arranged around the four operations that turn raw information into a graph-powered assistant: construction from text, embedding of the resulting structure, semantic retrieval against it, and the special case of already-structured sources like websites and contracts.

Chapter Two

Creating a Graph from Text

How GraphRAG replaced the NER pipeline with a single LLM pass

A research paper is a graph pretending to be prose. The sentence "CRISPR-Cas9 cleaves double-stranded DNA at the PAM recognition site" asserts three things about two entities connected by a verb. The construction pipeline's job is to stop pretending and extract the structure explicitly — but how that extraction happens changed fundamentally with the arrival of GraphRAG.

A brief history — before and after GraphRAG

Before GraphRAG (pre-2023), constructing a knowledge graph from text required a chain of specialised models: a NER tagger to label entity spans, a coreference resolver to cluster "Cas9 / it / the enzyme" into one identity, a supervised relation-extraction classifier trained on labelled pairs, and a separate deduplication step. Each model had its own failure modes; errors compounded across the chain.

GraphRAG (Microsoft Research, 2024) replaced all of that with a single observation: a sufficiently capable LLM can do all of it in one prompt per chunk — detect entities, resolve coreference, name the relation, and output structured triples simultaneously. The separate NER model is gone. The coref resolver is gone. The RE classifier is gone. One LLM call does what four models used to do, and it generalises to any domain without retraining.

§The GraphRAG Pipeline

Construction now has three stages, not five. The LLM collapses the middle three into a single structured generation step.

  1. Chunking — split the source text into overlapping windows of ~600 tokens so no fact straddles a context boundary.
  2. LLM extraction pass — send each chunk to the LLM with a prompt that asks it to simultaneously detect entities, resolve coreferences, and name the relationships between them. Returns structured JSON containing nodes and edges in one shot.
  3. Entity normalisation & graph merge — deduplicate equivalent nodes across chunks ("CRISPR/Cas9", "Cas9 nuclease", "SpCas9" → one canonical node), then write everything into the graph store.

§Pseudocode: GraphRAG Construction Algorithm

Below is the corrected construction loop. Notice that the inner loop calls the LLM once per chunk and receives back both entities and relations together — no separate NER step, no separate RE step.

Algorithm 1 — GraphRAG build_knowledge_graph(corpus)
──────────────────────────────────────────────────────────────
 INPUT  : corpus  — list of raw document strings
 OUTPUT : G       — graph (V nodes, E edges) + community index
                                                              
 KEY POINT: no NER model, no coref resolver, no RE classifier.
 The LLM does entity detection + coreference + relation       
 extraction in a SINGLE call per chunk.                       
──────────────────────────────────────────────────────────────

function build_knowledge_graph(corpus):

  G        ← empty_graph()
  registry ← EntityRegistry()   // canonical name → node_id

  ── Stage 1: Chunk every document ─────────────────

  all_chunks ← []
  for doc in corpus:
    chunks ← sliding_window(
      text     = doc,
      size     = 600,              // tokens per chunk
      overlap  = 100,             // overlap to catch cross-boundary facts
      strategy = "sentence_aware"  // never split mid-sentence
    )
    all_chunks.extend(chunks)

  ── Stage 2: LLM extraction pass — one call per chunk
     The LLM simultaneously:                         
     · identifies entities and their types           
     · resolves coreferences within the chunk        
     · extracts relationships between entities       
     · returns structured JSON — no post-processing  

  for chunk in all_chunks:

    extraction ← llm_extract(chunk.text)
    // Returns: { "entities": [...], "relations": [...] }
    // See Algorithm 1b for the exact prompt

    // Register entities from this chunk
    for ent in extraction.entities:
      node_id ← registry.lookup_or_create(
        name      = ent.name,
        type      = ent.type,
        threshold = 0.88           // fuzzy dedup across chunks
      )
      G.add_node_if_absent(node_id,
        attrs    = {type: ent.type, description: ent.description},
        source   = chunk.doc_id,
        chunk_id = chunk.id
      )

    // Write edges from this chunk
    for rel in extraction.relations:
      s ← registry.lookup(rel.source)
      o ← registry.lookup(rel.target)
      if s and o:
        G.add_edge(s, rel.predicate, o,
                   weight   = rel.weight,
                   source   = chunk.doc_id,
                   chunk_id = chunk.id)

  ── Stage 3: Community detection ──────────────────
     Group tightly connected nodes; LLM summarises   
     each group — these summaries answer global       
     "what is this corpus about?" queries.            

  communities ← leiden_algorithm(G)
  // Leiden: hierarchical, resolution-stable community detection

  for community in communities:
    nodes_text ← serialize_subgraph(community.nodes, G)
    community.summary ← llm_summarise(nodes_text)
    // LLM writes 2-3 sentence summary of what this community is about

  G.attach_communities(communities)

  return G
Stage 2 is the GraphRAG insight — one LLM prompt replaces three specialised models. Stage 3 (community detection + LLM summaries) is what makes the graph navigable at scale: rather than walking millions of edges, a query can match against a few hundred community summaries first.

§The Extraction Prompt — All-in-One

The single extraction prompt is what makes GraphRAG work. It asks the LLM to do entity detection, coreference resolution, and relation extraction simultaneously — returning one clean JSON object per chunk. No pre-fed entity list, no constrained schema required (though one can be added for domain-specific graphs).

Algorithm 1b — llm_extract(chunk_text) · the core prompt
── The single LLM call that replaces NER + coref + RE ───

SYSTEM = """
You are a knowledge graph extraction engine.
Given a text passage, return ONLY valid JSON — no explanation.

Your job (all in one pass):
  1. Detect every named entity: people, organisations, concepts,
     chemicals, locations, events, technologies, etc.
  2. Resolve coreferences: if "it", "the enzyme", "the company"
     refer to a previously mentioned entity, use that entity's
     canonical name — do NOT create a new node.
  3. Extract every meaningful relationship between entity pairs.
  4. Assign a weight 0.0–1.0 to each relation (your confidence).

Output schema (JSON, nothing else):
{
  "entities": [
    {
      "name":        "canonical name (use the most complete form)",
      "type":        "PERSON | ORG | CONCEPT | CHEMICAL | PLACE | EVENT | TECH",
      "description": "one sentence defining this entity in context"
    }
  ],
  "relations": [
    {
      "source":    "entity name (must appear in entities list)",
      "predicate": "verb phrase describing the relationship",
      "target":    "entity name (must appear in entities list)",
      "weight":    0.0-1.0
    }
  ]
}
"""

USER = chunk_text   // just the raw chunk — no pre-processing

The LLM reads "It was co-developed by…" and correctly resolves the pronoun to CRISPR-Cas9 without a separate coreference model — this is the core capability shift. The output is directly usable as graph nodes and edges.
Why provenance still matters

Even though extraction is now one LLM call, every node and edge still carries its doc_id and chunk_id. When the chatbot answers a question it can trace every claim back to the exact passage it came from — trustworthiness the Turing Institute identifies as one of the hardest open challenges in KG construction at scale.

Chapter Three

Embedding the Graph

Giving every node a position in vector space

A graph is a topology — it tells you who is adjacent to whom. An embedding is a geometry — it tells you how close is this node to that one. The two representations are complementary. Graph traversal is exact but brittle; vector similarity is fuzzy but generalises to unseen terms. A good knowledge system needs both, so the embedding step converts the graph into vectors while preserving the structural information the graph contains.

§Three Strategies for Graph Embedding

StrategyWhat it encodesBest for
TransE / TransRRelation as translation in embedding space: h + r ≈ tFact prediction; link completion; sparse graphs
Node2Vec / DeepWalkCo-occurrence on random walks; preserves neighbourhood structureSimilarity search; community detection
GraphSAGE / GNNAggregate neighbour features iteratively; rich attribute graphsEntity classification; retrieval-augmented generation

For a retrieval application (our main goal), the most practical approach combines a language model's text encoding of the node's label and description with a structural signal from the graph topology. The pseudocode below implements this hybrid.

§Pseudocode: Embedding Algorithm

Algorithm 2 — embed_knowledge_graph(G)
──────────────────────────────────────────────────────────────
 INPUT  : G       — knowledge graph (V nodes, E edges)
 OUTPUT : index   — vector index; node_id → embedding vector
──────────────────────────────────────────────────────────────

function embed_knowledge_graph(G, text_model, walk_config):

  embeddings ← {}          // node_id → vector

  ── Part A: Text-based node embedding ─────────────
  for v in G.nodes:

    // Build a rich textual description of the node
    description ← node_to_text(v)
    // e.g. "CRISPR-Cas9 [CONCEPT]: a molecular tool that
    //  cleaves DNA. Discovered by: Doudna, Charpentier.
    //  Used_in: gene editing, therapy."

    text_emb ← text_model.encode(description)
    // shape: [d_text]  e.g. 768 or 1536 dimensions

  ── Part B: Structural embedding via random walks ──
  walk_corpus ← []

  for v in G.nodes:
    for _ in range(walk_config.walks_per_node):   // e.g. 10
      walk ← biased_random_walk(
        start     = v,
        length    = walk_config.walk_length,       // e.g. 80
        p         = walk_config.return_param,      // BFS bias
        q         = walk_config.inout_param        // DFS bias
      )
      walk_corpus.append(walk)

  // Train skip-gram on walk sequences (Word2Vec)
  struct_model ← Word2Vec(
    sentences  = walk_corpus,
    vector_size= walk_config.dim,   // e.g. 128
    window     = 5,
    min_count  = 1,
    workers    = 4
  )

  ── Part C: Fuse text + structural embeddings ──────
  for v in G.nodes:
    text_emb   ← get_text_emb(v)       // [d_text]
    struct_emb ← struct_model[v.id]      // [d_struct]

    // Option A: simple concatenation
    fused ← concat([text_emb, struct_emb])

    // Option B: learned projection (preferred)
    fused ← fusion_layer.forward(text_emb, struct_emb)
    // fusion_layer = Linear(d_text + d_struct → d_out=512)

    embeddings[v.id] ← l2_normalize(fused)

  ── Part D: Also embed edge predicates ──────────────
  for (s, pred, o) in G.edges:
    relation_emb ← text_model.encode(pred)  // e.g. "cleaves"
    embeddings[edge_key(s,pred,o)] ← relation_emb

  ── Part E: Build searchable vector index ───────────
  index ← VectorIndex.build(
    vectors   = embeddings,
    metric    = "cosine",
    backend   = "FAISS"        // or Weaviate, Pinecone…
  )

  return index, embeddings
The biased random walk parameters p and q (Node2Vec) control whether walks prefer to stay local (BFS, good for community structure) or explore outward (DFS, good for role similarity). Tuning these two numbers changes what "similar" means in the embedding space.

§The TransE Intuition

For link prediction tasks (completing missing edges) a simpler geometric model is often used. TransE represents each relation as a translation vector: if (head, relation, tail) is a true triple, then the embedding of the tail should be approximately equal to the embedding of the head plus the embedding of the relation.

Algorithm 2b — TransE training loop (simplified)
── TransE: learn h, r, t ∈ ℝ^d such that h + r ≈ t ────

function train_transe(G, dim=100, epochs=500, margin=1.0):

  // Initialise entity and relation embeddings uniformly
  E ← uniform_init(size=(|G.nodes|, dim),  range=(-6/√dim, 6/√dim))
  R ← uniform_init(size=(|G.relations|, dim), range=(-6/√dim, 6/√dim))

  for epoch in range(epochs):
    for (h, r, t) in random_batch(G.triples, size=128):

      // Sample a corrupted negative triple
      t_neg ← random_entity(G.nodes, exclude=t)
      // (replace tail; alternatively replace head)

      // Compute distances
      d_pos ← L1_distance(E[h] + R[r], E[t])
      d_neg ← L1_distance(E[h] + R[r], E[t_neg])

      // Margin-ranking loss: push positives close, negatives far
      loss ← max(0, margin + d_pos - d_neg)

      backprop(loss)
      normalise_rows(E)   // constrain entity norms to 1

  return E, R

── At inference: predict missing tail ─────────────────
// "What does CRISPR-Cas9 cleave?"
query_vec ← E["CRISPR-Cas9"] + R["cleaves"]
candidates ← index.nearest_neighbours(query_vec, k=10)
// → ["double-stranded DNA", "single-stranded RNA", ...]
TransE's elegance is that the arithmetic h + r = t directly encodes what are the likely tails of this (head, relation) pair as a nearest-neighbour search.

Chapter Four

Semantic Retrieval

Walking the graph in response to a question

Construction and embedding are one-time costs. Retrieval is what the user experiences every time they ask a question. The system must take natural language, locate the right region of the graph, and return a set of nodes and passages grounded enough for a language model to synthesise into a correct answer. It does this in three distinct phases: query parsing, graph traversal, and re-ranking.

Query parsing — LLM, not NER

At query time, GraphRAG also uses the LLM — not a NER tagger — to extract entity mentions from the user's question. This consistency matters: the same reasoning capability that built the graph is used to navigate it. A NER model trained on news might miss domain-specific terms like "PAM site" or "off-target indel"; the LLM that extracted those terms originally will recognise them at query time.

§The Retrieval Flow

§Pseudocode: Semantic Retrieval Algorithm

Algorithm 3 — semantic_retrieve(query, G, index)
──────────────────────────────────────────────────────────────
 INPUT  : query   — natural language question string
          G       — knowledge graph
          index   — hybrid vector index (from Algorithm 2)
 OUTPUT : context — ranked list of (node, passage, score)
──────────────────────────────────────────────────────────────

function semantic_retrieve(query, G, index, config):

  ── Phase 1: Parse the query ───────────────────────

  // A: embed the query as a vector
  q_vec ← text_model.encode(query)

  // B: extract entity mentions via LLM (not NER model)
  // Consistent with construction — same LLM recognises
  // domain terms it previously extracted into the graph
  q_entities ← llm_extract_query_entities(query)
  // prompt: "List every named entity in this question.
  //  Return JSON: [{name, type}]. Nothing else."
  // e.g. [{"name":"CRISPR-Cas9","type":"TECH"},
  //        {"name":"off-target effects","type":"CONCEPT"}]

  ── Phase 2: Dual lookup ───────────────────────────

  // Path A — vector similarity (handles paraphrase)
  vector_hits ← index.search(q_vec, k=config.top_k)
  // returns [(node_id, cosine_score), ...]

  // Path B — exact entity anchor lookup
  anchor_nodes ← []
  for ent in q_entities:
    node ← G.find_node(name=ent, fuzzy=True)
    if node: anchor_nodes.append(node)

  candidate_nodes ← union(vector_hits.ids, anchor_nodes)

  ── Phase 3: k-hop graph traversal ────────────────

  subgraph_nodes ← set(candidate_nodes)

  for node in candidate_nodes:
    for hop in range(config.max_hops):      // usually 1–2

      neighbours ← G.neighbours(node,
        relation_filter = config.allowed_relations,
        direction       = "both"           // in + out edges
      )

      for (neighbour, relation, weight) in neighbours:

        // Prune by relation relevance to query
        rel_emb   ← index.get_edge_emb(relation)
        rel_score ← cosine_sim(q_vec, rel_emb)

        if rel_score ≥ config.relation_threshold:   // e.g. 0.55
          subgraph_nodes.add(neighbour)

  ── Phase 4: Pull community summaries ─────────────

  communities ← set()
  for node in subgraph_nodes:
    communities.add(G.community_of(node))

  summaries ← [community.summary_text
               for community in communities]

  ── Phase 5: Fetch source passages + re-rank ──────

  passages ← []
  for node in subgraph_nodes:
    for chunk_ref in node.source_chunks():
      passage ← corpus.fetch(chunk_ref)
      passages.append(passage)

  // Score each passage against query (cross-encoder)
  scored ← [(p, cross_encoder.score(query, p)) for p in passages]

  // MMR: balance relevance with diversity
  final_passages ← maximal_marginal_relevance(
    scored,
    lambda_param = 0.6,   // 0 = max diversity, 1 = max relevance
    k = config.final_k    // top passages to return, e.g. 8
  )

  ── Phase 6: Compose prompt ────────────────────────

  context ← build_context(
    summaries  = summaries,       // high-level orientation
    passages   = final_passages,  // grounded evidence
    subgraph   = subgraph_nodes   // structural relationships
  )

  return context   // → feed to LLM for answer generation
MMR (Maximal Marginal Relevance) is the key to avoiding answer collapse — without it, the top-8 passages might all be slight variations of the same fact. MMR explicitly penalises redundancy.
The difference from plain RAG

A standard vector RAG runs only Phase 1 (embed) and the top-k of Phase 2 (vector hits), then skips directly to Phase 6. A graph-backed system adds Phases 3 and 4 — the traversal and community summaries. In the Turing group's framing, this is the shift from "data-driven retrieval" to "knowledge-enhanced data-driven retrieval."

Chapter Five

Websites & Contracts

When the source is already half a graph

A website has hyperlinks. A contract has clause references. A product catalogue has parent-child category relationships. These sources are already relational — they just hide their structure inside HTML anchors, numbered headings, and parenthetical cross-references. The job here is not extraction from noise, as with free text, but faithful harvesting of structure that is already present, followed by the same embedding and retrieval machinery from chapters three and four.

§Website Graph Construction

A website graph has three types of node — page nodes (each URL is a node), section nodes (headings within a page), and entity nodes (products, categories, people mentioned in the content). Their relationships come from three sources: hyperlinks (structural), heading hierarchy (containment), and NER on page content (semantic).

§Pseudocode: Website Graph Builder

Algorithm 4a — build_website_graph(seed_url)
──────────────────────────────────────────────────────────────
 INPUT  : seed_url  — starting URL of the website
 OUTPUT : G         — typed knowledge graph of the site
──────────────────────────────────────────────────────────────

function build_website_graph(seed_url, max_pages=500):

  G        ← empty_graph()
  visited  ← set()
  frontier ← Queue([seed_url])

  ── Stage 1: Crawl + parse ─────────────────────────

  while frontier not empty and |visited| < max_pages:

    url ← frontier.dequeue()
    if url in visited: continue
    visited.add(url)

    html ← fetch_page(url)
    dom  ← parse_html(html)

    // Create page node with metadata
    page_node ← Node(
      id       = url,
      type     = classify_page_type(url, dom),
      // e.g. PRODUCT | CATEGORY | SUPPORT | POLICY | BLOG
      title    = dom.title(),
      text     = dom.extract_main_text(),
      metadata = {"last_modified": dom.meta_date()}
    )
    G.add_node(page_node)

    ── Stage 2: Extract structural links ─────────────

    for link in dom.all_links():
      if same_domain(link.href, seed_url):
        // Determine relation type from context
        rel_type ← classify_link_relation(
          source_type = page_node.type,
          link_text   = link.anchor_text,
          href        = link.href
        )
        // e.g. "contains", "references", "related_to", "governed_by"
        G.add_edge(page_node.id, rel_type, link.href)
        frontier.enqueue(link.href)

    ── Stage 3: Extract section/heading hierarchy ────

    prev_section ← page_node
    for heading in dom.headings():   // h1, h2, h3…
      sec_node ← Node(
        id   = url_hash(url + heading.text),
        type = "SECTION",
        text = heading.extract_following_text(max_words=300)
      )
      G.add_node(sec_node)
      G.add_edge(page_node.id, "has_section", sec_node.id)
      if heading.level > prev_section.level:
        G.add_edge(prev_section.id, "parent_of", sec_node.id)
      prev_section ← sec_node

    ── Stage 4: Entity enrichment on page content ─────
    // For websites, NER is actually still reasonable here —
    // pages are short and the LLM overhead per-page is high.
    // A lightweight NER (spaCy) or an LLM batch call both work.

    entities ← extract_entities(page_node.text)
    // either: ner_model.predict()  — fast, good for product/place names
    // or:     llm_extract()        — better for abstract concepts
    for ent in entities:
      ent_node ← registry.lookup_or_create(ent)
      G.add_edge(page_node.id, "mentions", ent_node.id)

  return G
The classify_link_relation function is where most of the intelligence lives. A link from a Category page to a Product page is "contains"; a link from a Product page to the Returns Policy is "governed_by". These distinctions are what make the chatbot precise.

§Contract Graph Construction

Contracts are even more structured than websites. They have a formal anatomy — parties, effective dates, clauses, obligations, and cross-references — and most of that structure is recoverable from layout alone (numbered sections, capitalised defined terms, "as defined in Clause X" patterns).

§Pseudocode: Contract Parser + Chatbot Query

Algorithm 4b — parse_contract(pdf_path)
──────────────────────────────────────────────────────────────
 INPUT  : pdf_path  — path to a contract document (PDF)
 OUTPUT : G         — obligation knowledge graph
──────────────────────────────────────────────────────────────

function parse_contract(pdf_path):

  G ← empty_graph()

  text   ← pdf_to_text(pdf_path, preserve_layout=True)
  blocks ← segment_by_clause(text)
  // detect numbered clauses: "3.1", "4.2", etc.

  ── Stage 1: Extract parties and dates ─────────────

  header_block ← find_header(text)
  parties ← extract_parties(header_block)
  // regex + NER to find "between X (the Client) and Y (the Provider)"
  dates   ← extract_key_dates(header_block)

  for party in parties:
    G.add_node(Node(id=party.name, type="PARTY",
                    role=party.role, attrs=party.details))
  for d in dates:
    G.add_node(Node(id=d.label, type="DATE", value=d.value))

  ── Stage 2: Parse clause structure ───────────────

  for clause in blocks:

    clause_node ← Node(
      id    = clause.number,   // e.g. "CL_4.2"
      type  = "CLAUSE",
      title = clause.heading,
      text  = clause.body
    )
    G.add_node(clause_node)

    // Detect which parties are bound by this clause
    bound ← find_binding_parties(clause.body, parties)
    for party in bound:
      G.add_edge(party.name, "bound_by", clause.number)

    // Cross-references to other clauses
    refs ← extract_clause_refs(clause.body)
    // regex: "as defined in Clause 3.1", "subject to Clause 9"
    for ref in refs:
      G.add_edge(clause.number, "references", ref.target)

  ── Stage 3: Extract obligations via LLM ──────────

  obligation_prompt ← """
  From the clause below, extract all obligations.
  Return JSON: [{
    "obligor": party name,
    "action":  verb phrase,
    "object":  what they must do,
    "condition": any condition (or null),
    "deadline": any date/period (or null)
  }]
  Clause: {clause_text}
  """

  for clause in blocks:
    obligations ← llm_call(obligation_prompt.format(
      clause_text=clause.body))

    for ob in obligations:
      ob_node ← Node(id=uuid(), type="OBLIGATION",
                       action=ob.action, object=ob.object,
                       condition=ob.condition, deadline=ob.deadline)
      G.add_node(ob_node)
      G.add_edge(clause.number, "creates", ob_node.id)
      G.add_edge(ob_node.id, "owed_by", ob.obligor)

  return G
The obligation extraction uses an LLM in structured-output mode, while the clause cross-references come from simple regex — the two approaches are complementary. Regex is fast and deterministic; LLM reasoning handles the nuance.

§Pseudocode: Chatbot Query over Structured Graph

Algorithm 4c — answer_contract_query(query, G)
──────────────────────────────────────────────────────────────
 INPUT  : query — natural language question about a contract
          G     — obligation knowledge graph (from 4b)
 OUTPUT : answer with clause citations
──────────────────────────────────────────────────────────────

function answer_contract_query(query, G):

  ── Step 1: Classify intent ────────────────────────

  intent ← classify_query_intent(query)
  // Intent types and example queries:
  // OBLIGATION_LOOKUP → "What must Acme pay?"
  // PARTY_LOOKUP      → "Who is the service provider?"
  // TERMINATION       → "Can the contract be ended early?"
  // DATE_LOOKUP       → "When does this expire?"
  // CROSS_REF         → "What does Clause 7 require?"

  ── Step 2: Extract anchor entities ───────────────

  anchors  ← []
  // Use LLM to extract entity mentions from the query
  entities ← llm_extract_query_entities(query)
  // e.g. [{name:"Acme", type:"PARTY"}, {name:"terminate", type:"ACTION"}]

  for ent in entities:
    node ← G.find_node(ent, type_hint=intent.expected_type())
    if node: anchors.append(node)

  ── Step 3: Intent-specific graph query ────────────

  subgraph ← empty_graph()

  if intent == OBLIGATION_LOOKUP:
    for party_node in anchors:
      clauses ← G.traverse(party_node, via="bound_by")
      for clause in clauses:
        obs ← G.traverse(clause, via="creates")
        add_to_subgraph(subgraph, party_node, clause, obs)

  elif intent == TERMINATION:
    term_clauses ← G.find_nodes(type="CLAUSE",
                                  attr_contains={"title": "terminat"})
    for cl in term_clauses:
      refs    ← G.traverse(cl, via="references")
      parties ← G.traverse(cl, via="bound_by", reverse=True)
      add_to_subgraph(subgraph, cl, refs, parties)

  elif intent == DATE_LOOKUP:
    dates ← G.find_nodes(type="DATE")
    add_to_subgraph(subgraph, dates)

  // (other intents handled similarly)

  ── Step 4: Serialise subgraph to context ─────────

  context_lines ← []
  for node in subgraph.nodes:
    context_lines.append(node_to_prose(node))
    // e.g. "Clause 4.2 (Payment Terms) creates obligation:
    //       Acme Ltd must pay £12,000 net 30 days."
  for (s, rel, o) in subgraph.edges:
    context_lines.append(edge_to_prose(s, rel, o))

  ── Step 5: LLM synthesis ──────────────────────────

  prompt ← build_prompt(
    system  = "Answer using ONLY the provided contract context.",
    context = join(context_lines),
    query   = query
  )
  answer ← llm.generate(prompt)

  // Attach citations to the answer
  citations ← [node.id for node in subgraph.nodes
               if node.type == "CLAUSE"]
  return {"answer": answer, "cited_clauses": citations}

//    may terminate with 60 days written notice, provided

//    Terms) are first settled.",

The intent classifier is the router. Different intent types trigger different traversal patterns — obligation lookup walks "bound_by → creates", termination lookup finds clause nodes by attribute, date lookup is a direct node type filter. Each traversal is deterministic and auditable.

For a website-backed assistant, the flow is similar but the anchor entity is typically a product or category rather than a clause. The chatbot walks the site graph instead of a contract graph.

Chapter Six

One Pattern, Many Surfaces

What all four operations share

Read end to end, the four algorithms in this guide describe a single principle repeated across different source materials: find the things, name the relationships between them, give each thing a position in geometry, and use that position plus the structure together when a question arrives. The source material changes — a research paper, a website, a contract — but the pattern does not.

Embedding Text embed (node description) + Node2Vec (structure) Page embedding + structural position (depth, in-degree) Clause embedding + obligation vectors for semantic lookup
Retrieval Dual lookup → k-hop traversal → community summary → MMR re-rank Intent → product node → governed_by / helps_with traversal Intent classifier → obligation / termination traversal → LLM
Key advantage vs. plain RAG Cross-document reasoning; explainable citations Deterministic path: no hallucinated page references Structured obligations; auditable clause citations

The Turing Institute's description of knowledge graphs as alive points to the most compelling practical property: none of these graphs are static artefacts. A website changes; a contract is amended; new research papers are published. Because the graph stores provenance, incremental updates are possible — add a node, retire an edge, re-embed the changed neighbourhood — without rebuilding from scratch.

Turing Institute · The larger vision

Modern KGs are the result of complex assemblies of manual and automatic modelling and data ingestion pipelines. Recent research looks at trade-offs between semantic representations and deep learning, and more broadly at designing symbolic/connectionist AI architectures — exactly the hybrid described in this guide.

Applications span healthcare data governance, financial fraud detection, legal knowledge management, scientific discovery, and the intelligent systems that let engineers and scientists reason over large corpora they could never read in full.

The closing thought is the simplest one. Knowledge has shape. When we represent it as text, that shape is implicit and must be reconstructed on every read. When we represent it as a graph, the shape is given — written down, queryable, portable, auditable. The algorithms in this guide are the machinery for making that shape explicit. The graph does not replace the text; it is what the text was saying, finally made visible.
